# Multi-Agent Consult v1

## Summary
Build a local, single-user Next.js full-stack app on Node 20 with `npm`, TypeScript, Tailwind, Drizzle, and SQLite. v1 implements one provider adapter against OpenRouter, but the orchestration, tool broker, and model catalog layers are provider-pluggable so direct OpenAI Codex OAuth and Anthropic OAuth can be added later without redesigning the run engine.

Each run has two participant models and one configurable referee. Participants can use web research tools and optional read-only workspace tools, then exchange proposals and critiques until the referee declares convergence or the max-turn limit is reached. Participants may also submit candidate user questions, but only the referee can issue user-facing question batches. Runs stream live, persist to SQLite, and produce a copy-paste-friendly final answer with a separate `Sources` section.

Implementation must create and maintain:
- `docs/multi-agent-consult.plan.md`: this plan.
- `docs/multi-agent-consult.impl.md`: a living log with `Decisions`, `Progress`, `Gaps`, `Blockers`, and `Verification`, updated after each meaningful implementation step.

## Key Changes
### App and storage
- Use Next.js App Router with route handlers and SSE for live updates.
- Use Drizzle with SQLite for persisted `runs`, `participants`, `turns`, `tool_invocations`, `source_records`, `referee_decisions`, and `user_question_batches`.
- Keep the app local-first and single-user. Allow many historical runs, but only one active run at a time in v1.
- Accept provider credentials from server env only:
  - `OPENROUTER_API_KEY` required
  - `BRAVE_SEARCH_API_KEY` optional, required only when the app-owned search backend is selected

### Provider and capability layer
- Define a `ProviderAdapter` interface with `listModels`, `validateRoleCapabilities`, `createChatStream`, `supportsProviderNativeSearch`, and `normalizeSources`.
- Implement `OpenRouterAdapter` first and populate model pickers from the live OpenRouter catalog.
- Validate model roles before a run starts:
  - participant models must support tool calling
  - referee model must support structured JSON output
  - provider-native search can only be selected when both participant models and the adapter can normalize returned citations into the app’s `SourceRecord` format
- Keep provider selection internal for v1, but structure adapters so future `OpenAIAdapter` and `AnthropicAdapter` can reuse the same run engine and tool interfaces.

### Tool broker and run configuration
- Split tools into two independent groups in the run builder:
  - `Web research`: `off`, `provider_native`, or `brave`
  - `Workspace context`: `off` or `path-based`
- Web research is first-class in v1:
  - `web_search(query, searchType?)`
  - `fetch_web_page(url)`
- Workspace tools are optional and read-only:
  - `list_files`
  - `search_files`
  - `read_file`
  - `run_check`
- `run_check` is not arbitrary shell. It exposes only discovered non-mutating commands such as `git_status`, `git_diff_stat`, and repo scripts like `build`, `lint`, `test`, `typecheck` when present.
- Workspace tools are scoped to a validated absolute root path. All file access and discovered commands must stay inside that root.
- No file edits, browser automation, or arbitrary shell in v1.

### Debate, referee, and user-question workflow
- Implement a `DebateCoordinator` state machine with this flow:
  1. Create both participant contexts from the shared task prompt, optional participant personas, selected tool groups, and optional workspace manifest.
  2. Turn 0: both participants produce independent initial proposals in parallel.
  3. Referee evaluates both outputs using a strict JSON schema.
  4. If converged, synthesize the final consensus and stop.
  5. If not converged, participants receive opponent output plus referee guidance and produce revised proposals in parallel.
  6. Repeat until converged, cancelled, or max turns reached.
- Referee JSON schema must include: `converged`, `confidence`, `summary`, `preferredDraft`, `requiredNextFocus`, `remainingDisagreements`, `needsUserInput`, and optional `questionBatch`.
- Add a participant-only tool for clarification proposals: `propose_user_questions`.
- Participants can submit candidate questions, but those are never shown directly to the user. The referee reviews, merges, rewrites, or discards them.
- When the referee decides clarification is needed, it emits one `UserQuestionBatch` containing 1-3 questions.
- Each question is rendered in the UI one at a time, planning-mode style, with:
  - multiple-choice options
  - a clearly marked recommended option
  - an optional note field
- The run pauses until the batch is answered or skipped. The full answer batch is then appended to run context for both participants before the next debate round.
- Final synthesis is generated by the referee model from the last participant drafts plus referee history. The final output format is:
  - `Solution`
  - `Rationale`
  - `Sources`
- Citations are required for web-derived claims, but they appear in the trailing `Sources` section or transcript source panels, not inline inside the solution body.

### UI and operator experience
- Main run builder includes:
  - participant model A
  - participant model B
  - referee model
  - shared task prompt
  - optional participant personas
  - optional referee persona
  - max turns
  - web research mode
  - optional workspace path
- Transcript view streams participant turns, tool activity, referee decisions, pause/resume state, user question batches, and final synthesis.
- Show source links as separate structured cards or a `Sources` block below each completed turn, not mixed into the prose.
- History view lists persisted runs and lets the user reopen transcripts and final outputs.
- Show token and latency metrics after each response, but do not enforce token or dollar budget caps in v1.

## Test Plan
- Unit test the debate state machine for:
  - immediate convergence after turn 0
  - multi-round convergence
  - max-turn exhaustion
  - user-question pause and resume
  - cancellation while paused or streaming
- Unit test tool guards for:
  - out-of-root file access rejection
  - unsupported `run_check` commands
  - missing Brave API key when `brave` search is selected
  - invalid provider-native-search model combinations
- Integration test search behavior for:
  - Brave-backed `web_search` plus `fetch_web_page`
  - provider-native search with normalized citations
  - citation capture into `SourceRecord`
- Integration test SSE ordering for participant turns, referee decisions, question pauses, answer resume, and final synthesis.
- End-to-end test the main flows:
  - web-research-only run
  - workspace-assisted coding run
  - referee-issued clarification batch
  - persisted history reopen

## Assumptions and Defaults
- Package manager is `npm`.
- v1 is local self-hosted only.
- OpenRouter is the only implemented model provider in v1.
- Web research is enabled by default when a backend is available.
- Default search backend is `brave` when `BRAVE_SEARCH_API_KEY` is configured; otherwise offer `provider_native` only when supported by the selected models.
- Workspace tools are optional and disabled when no workspace path is supplied.
- No mutating tools ship in v1.
- Stop reasons are only `converged`, `max_turns`, `user_cancelled`, or `failed`.
- `docs/multi-agent-consult.impl.md` must be updated continuously during implementation with timestamped decisions, progress, open gaps, blockers, and verification notes.
